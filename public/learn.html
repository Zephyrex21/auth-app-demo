Authentication vs Authorization

Two words people mix up all the time. Short answer: AuthN = Who are you?  ·  AuthZ = What can you do?

Authentication (AuthN)

Proving you are who you say you are. Like showing your ID at a club entrance. Email + password → server checks → you're authenticated.

Authorization (AuthZ)

Once you're in — what are you allowed to do? An admin can delete things. A regular user can't. Different people, different permissions.

In This App (AuthN)

You type email + password → server checks them → hands you a JWT token. That whole process is authentication.

In This App (AuthZ)

The protect middleware reads your token before any locked route. No valid token = 401 error. That's authorization enforcing your permission.

JSON Web Tokens (JWT)

A JWT is a signed ticket the server gives you after login. You carry it in your browser and show it on every protected request. Three parts separated by dots.
// Three base64url-encoded parts, separated by dots eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9 // Header . eyJpZCI6IjY0YWJjLi4uIiwiZW1haWwiOiJ5b3VAZXhhbXBsZS5jb20ifQ // Payload . SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c // Signature

Header

Just says which algorithm was used to sign the token (HS256). Base64 encoded — not encrypted. Anyone can read it.

Payload (Claims)

Your actual data: user ID, email, when it was issued, when it expires. Also not encrypted — never put passwords or secrets in here.

Signature

HMAC stamp of (header + payload) using your server's secret key. If anyone edits even one character of the token, this breaks and the server rejects it.

Stateless

The server doesn't store anything about you. Your token IS the proof. Multiple servers can all verify the same token — no shared database needed.

Password Hashing with bcrypt

bcrypt scrambles your password into random-looking gibberish that can never be reversed. Even if your database gets hacked, attackers just see nonsense.
// In authController.js const salt = await bcrypt.genSalt(10); // 2^10 = 1024 rounds const hash = await bcrypt.hash(password, salt); // "abc123" → "$2a$10$..." // On login — compare entered password vs stored hash const isMatch = await bcrypt.compare(entered, storedHash);

Why not MD5 / SHA256?

They're too fast — modern GPUs can try billions of password guesses per second. bcrypt is intentionally slow, making that impractical.

Salt

A random string mixed into your password before hashing. So even if two users have the same password, their stored hashes look completely different. Defeats pre-computed attack tables.

Cost Factor

Cost=10 means 1024 hashing rounds. Each round makes brute-force attacks slower. Higher cost = safer but slower login. 10–12 is the sweet spot.

One-way

There's no 'decrypt'. When you log in, bcrypt re-hashes what you typed and compares the two results. If they match — you're in. No reversal needed.

Sessions vs Token-based Auth

Two ways to keep users logged in after they authenticate. This app uses JWT tokens. Here's why that matters.

Session-based

Server keeps a record of every logged-in user in a database. On each request, it looks you up. Works fine for a single server, but hard to scale.

Token-based (JWT)

Server gives you a token and forgets about you. Every request proves itself with the token. Zero database lookup needed — just signature verification.

Session Pros

Easy to instantly log someone out (just delete their session). Server has full control. Great for sensitive apps like banking.

JWT Pros

Scales easily across multiple servers — no need to share session state. Perfect for APIs, mobile apps, and microservices.

Auth Middleware — How Protected Routes Work

Middleware is just a function that runs before your route handler. Think of the protect function as a bouncer checking IDs at the door. Valid token? Come in. Expired or missing? 401, goodbye.
// middleware/authMiddleware.js — exact code from this app const protect = (req, res, next) => { // 1. Extract token from "Authorization: Bearer <token>" header const authHeader = req.headers.authorization; if (!authHeader || !authHeader.startsWith("Bearer ")) { return res.status(401).json({ message: "No token provided." }); } const token = authHeader.split(" ")[1]; // everything after "Bearer " // 2. Verify the token's signature using the secret try { const decoded = jwt.verify(token, process.env.JWT_SECRET); req.user = decoded; // { id, email } — attached for controllers to use next(); // valid token → proceed to controller } catch (error) { return res.status(401).json({ message: "Invalid or expired token." }); } };

Why next()?

Express routes are a chain. Middleware calls next() to pass control forward. If it doesn't call next(), the request stops right there.

req.user

After verifying the JWT, the middleware attaches your user data to req.user. So the controller can get your user ID without hitting the database again.

Route Registration

protect is listed before getProfile in the route. Express runs them in order. If protect rejects the request, getProfile never even runs.

jwt.verify()

Throws an error if the token is expired, has been modified, or was signed with a different secret. The catch block turns that into a clean 401 response automatically.

Common Auth Vulnerabilities

Knowing what can go wrong is the first step to building securely. These are the most common auth attacks and what stops them.

XSS

Malicious JavaScript injected into a page can steal your localStorage token. Fix: use HttpOnly cookies (JS can't touch them) and always sanitize user input.

CSRF

A fake website tricks your browser into making requests using your cookies. Fix: set SameSite=Strict on cookies so they don't go cross-site.

JWT None Alg

Attackers change the JWT header to alg: none, hoping the server skips signature checking. Fix: always whitelist allowed algorithms on the server.

Brute Force

Bots try millions of password combinations automatically. bcrypt slows each attempt down, but you should also add rate limiting and account lockout in production.

Bearer Token Pattern

How the client sends its JWT with every protected request. You put it in a special header called Authorization. One line of code, every API call.
// Every API call to a protected route const res = await fetch('/api/users/profile', { method: 'GET', headers: { 'Authorization': `Bearer ${localStorage.getItem('token')}`, 'Content-Type': 'application/json' } });

Why "Bearer"?

It's a naming convention (RFC 6750). "Bearer" just means "whoever has this token is authorized." No username needed — the token is the key.

Header vs Cookie

Cookies are sent automatically by the browser (CSRF risk). Headers must be set manually in JS. Using a header means you control exactly when and where the token is sent.

Auth Glossary

Quick-reference definitions for every term in this app.
JWT (JSON Web Token)
A signed ticket the server gives you after login. Three parts separated by dots: header, payload, signature. The server can verify it without any database lookup.
bcrypt
A one-way password scrambler designed to be intentionally slow. It uses a salt and a cost factor to make brute-force cracking take years instead of seconds.
Salt
A random string mixed in before hashing. Makes sure two users with the same password get different hashes in the database, so pre-computed attack tables are useless.
Bearer Token
The convention for sending JWTs in API requests. You put the token in the Authorization header with the word "Bearer" in front. Whoever holds the token gets access.
Middleware
Code that runs before your route handler. In Express.js, middleware functions have access to req and res and can either pass control forward (next()) or stop the request.
iat (Issued At)
A standard JWT claim. A Unix timestamp (seconds since 1970) saying exactly when the token was created.
exp (Expiration)
A standard JWT claim. Unix timestamp after which the token is expired and must be rejected. This app sets it to 1 day (86400 seconds) after login.
HttpOnly Cookie
A cookie that JavaScript cannot read — only the browser sends it automatically. Set with the HttpOnly flag in the Set-Cookie header. The most secure place to store auth tokens.
HMAC-SHA256
The signing algorithm this app uses for JWTs. It combines your secret key with SHA-256 hashing to produce a tamper-proof signature. Standard and widely trusted.
RBAC (Role-Based Access Control)
Instead of giving permissions to each user individually, you assign them to roles (admin, editor, viewer) and then assign users to roles. Much easier to manage at scale.
Refresh Token
A second, longer-lived token used to silently get a new access token when yours expires — so users don't get randomly logged out. Not in this demo, but used in production apps.
401 vs 403
401 = you're not logged in (no valid token). 403 = you're logged in but not allowed to do this. Same word "unauthorized" gets thrown around for both — don't mix them up.
